## In Depth

### Reading leaked Claude Code source code

Claude Code's source code leaked this week, and buried inside is one of the most delightful Easter eggs in recent memory: a full-fledged companion pet system that almost nobody knew existed. The feature lives in a `src/buddy/` directory and generates a deterministic ASCII art companion based on your user ID—complete with species, rarity tiers, stats, animations, hats, and even a 1% chance of being shiny like a Pokémon.

Your companion is chosen from 18 species including duck, goose, cat, dragon, octopus, capybara, cactus, and the mysteriously named "chonk." Rarity follows classic gacha mechanics: 60% common, 25% uncommon, 10% rare, 4% epic, and 1% legendary. Rarity determines whether your buddy gets a hat—common companions go bareheaded, while higher tiers can sport crowns, wizard hats, halos, or even a "tinyduck" perched on top. Each companion has five D&D-style stats (DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK) with one peak stat and one dump stat.

The architecture cleverly splits "bones" (deterministic attributes regenerated from your user ID) and "soul" (name and personality generated once by the model and stored). This means you can't hack your config file to fake a legendary rarity—it's locked to your ID using Mulberry32, a tiny PRNG with the beautiful comment "good enough for picking ducks." The sprites are 5x12 ASCII grids with three animation frames each: cats wiggle their tails, dragons breathe tildes, and octopuses alternate tentacle positions.

The companion sits beside the input box and occasionally comments via speech bubble. The system prompt explicitly tells Claude not to pretend to be the companion or narrate what it might say, and if you address your buddy by name, Claude is instructed to "stay out of the way: respond in ONE line or less." The entire feature is hidden behind a `BUDDY` flag, making it one of the tech industry's best-kept secrets until now.

### JWT vs Session vs OAuth2 in Spring Boot: Which One Should You Use?

After eight years building Java backends, one developer has cut through the noise to deliver an honest comparison of the three main authentication approaches in Spring Boot: sessions, JWT, and OAuth2. The guide skips the usual "it depends" hand-waving and offers clear direction based on real production experience.

Session-based auth stores user state server-side, creating a session after login and sending a session ID via cookie. It's simple, battle-tested, and perfect for monoliths with server-side rendering (Thymeleaf, JSP). Spring Security handles it beautifully out of the box. The main pain point is horizontal scaling—multiple instances need shared session storage via Redis or Spring Session, adding infrastructure complexity.

JWT shines in REST APIs and microservices where stateless backends serve React/Angular/Vue frontends or mobile clients. Tokens travel across services without shared state. But there's a critical gotcha: token revocation. Once issued, a JWT remains valid until expiration. If you need immediate invalidation (compromised account, role changes), you're stuck unless you maintain a blocklist—which reintroduces server-side state. Most teams solve this with short-lived access tokens (15 minutes) plus longer-lived refresh tokens stored server-side, though this adds implementation complexity.

OAuth2 is a delegation protocol, not a replacement for sessions or JWT—it's a framework that often uses JWT under the hood. Use it when you need single sign-on, third-party authentication (Google, GitHub, Keycloak), or token-based service delegation. The guide includes production-ready Spring Security configuration examples for each approach, with clear recommendations: sessions for monoliths where simplicity matters, JWT for distributed systems prioritizing scalability, and OAuth2 when you need external authentication providers or sophisticated authorization flows.

*Source: [Dev.to Java](https://dev.to/springcraft/jwt-vs-session-vs-oauth2-in-spring-boot-which-one-should-you-use-389f)*

### Spring AI 2.0.0-M4, 1.1.4 and 1.0.5 are available now

The Spring AI team has shipped three releases simultaneously—versions 2.0.0-M4, 1.1.4, and 1.0.5—delivering 51 combined improvements, bug fixes, and documentation updates across all release streams. These updates bring important stability enhancements and new capabilities to developers integrating AI into Spring Boot applications.

The 1.0.5 maintenance release focuses on stability with eight bug fixes addressing vector store and streaming issues. Version 1.1.4 adds a notable new feature: dynamic structured output control, giving developers finer-grained control over model responses, alongside 11 bug fixes spanning vector stores, model integrations, and caching.

The milestone 2.0.0-M4 release packs the most changes with 28 improvements total. Key additions include enhanced Google Search integration for Gemini 3.x models and the same dynamic structured output control feature from 1.1.4. Importantly, this release includes deprecation notices for three model providers: Vertex AI, ZhiPu AI, and OCI GenAI integrations will be removed in future releases, and users should plan migration to alternative providers. Dependency updates include Google GenAI SDK 1.44.0, OpenAI SDK 4.28.0, and Anthropic SDK 2.17.0.

All three releases are available now from Maven Central. The Spring AI team continues building on these foundations with upcoming versions promising enhanced capabilities and improved developer experience. For anyone building AI-powered Spring Boot applications—whether you're adding LLM chat, semantic search with vector stores, or structured data extraction—these releases deliver production-ready stability and forward momentum toward Spring AI 2.0.

*Source: [Spring Blog](https://spring.io/blog/2026/03/26/spring-ai-2-0-0-M4-and-1-1-4-and-1-0-5-available)*
